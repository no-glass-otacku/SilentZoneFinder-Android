# 1. 데이터 저장 및 동기화

## 📌 데이터 저장 위치 관련 질문

### 1. 검색 기록 저장 방식
**Q: 검색 기록이 로컬 저장이면 로그아웃했을 때 여전히 검색 기록이 남아있나요?**
- **답변**: 네, 검색 기록은 `SharedPreferences`에 기기별로 저장됩니다. 로그아웃해도 해당 기기의 검색 기록은 유지됩니다. 다만, 사용자별로 분리되어 저장되므로 로그인한 사용자와 비로그인 사용자의 검색 기록은 별도로 관리됩니다.

**Q: 다른 기기에서 해당 계정으로 로그인 했을 때 검색 기록이 남아있나요?**
- **답변**: 아니요, 검색 기록은 서버에 저장되지 않고 기기 로컬(`SharedPreferences`)에만 저장되므로, 다른 기기에서는 검색 기록을 볼 수 없습니다. 각 기기마다 독립적으로 검색 기록이 관리됩니다.

**Q: 검색 기록을 서버에 저장하지 않는 이유는?**
- **답변**: 검색 기록은 개인적인 편의 기능이며, 기기별로 다를 수 있는 정보이기 때문에 로컬 저장이 더 적합합니다. 또한 서버 저장 시 개인정보 보호 이슈와 저장 공간 비용을 고려하여 로컬 저장 방식을 선택했습니다.

---

### 2. 리뷰 데이터 저장 방식
**Q: 리뷰는 어디에 저장되나요?**
- **답변**: 리뷰는 Supabase 데이터베이스의 `reviews` 테이블에 저장됩니다. 서버에 저장되므로 모든 사용자가 공유하며, 다른 기기에서도 동일한 계정으로 로그인하면 작성한 리뷰를 확인할 수 있습니다.

**Q: 리뷰 작성 시 오프라인 상태면 어떻게 되나요?**
- **답변**: 현재는 오프라인 상태에서 리뷰를 작성하면 저장에 실패합니다. 오프라인 캐싱 및 동기화 기능은 아직 구현되지 않았으며, 향후 개선 예정입니다.

**Q: 리뷰 수정/삭제 권한은 어떻게 관리하나요?**
- **답변**: Supabase의 RLS(Row Level Security) 정책을 통해 관리합니다. 사용자는 자신이 작성한 리뷰만 수정/삭제할 수 있으며, 다른 사용자의 리뷰는 읽기만 가능합니다.

---

### 3. 즐겨찾기 저장 방식
**Q: 즐겨찾기는 어디에 저장되나요?**
- **답변**: 즐겨찾기는 Supabase 데이터베이스의 `favorites` 테이블에 저장됩니다. 서버에 저장되므로 다른 기기에서도 동일한 계정으로 로그인하면 즐겨찾기 목록을 확인할 수 있습니다.

**Q: 로그아웃 후에도 즐겨찾기가 유지되나요?**
- **답변**: 즐겨찾기는 서버에 저장되므로, 로그아웃 후 다시 로그인하면 이전에 추가한 즐겨찾기가 그대로 남아있습니다. 다만, 비로그인 상태에서는 즐겨찾기 기능을 사용할 수 없습니다.

---

### 4. 알림 기록 저장 방식
**Q: 알림 기록은 어디에 저장되나요?**
- **답변**: 알림 기록은 `SharedPreferences`에 기기별로 저장됩니다. 서버에 저장되지 않으므로 다른 기기에서는 알림 기록을 확인할 수 없습니다.

**Q: 알림 기록을 서버에 저장하지 않는 이유는?**
- **답변**: 알림 기록은 기기별로 다를 수 있는 정보이며, 개인적인 히스토리 정보이기 때문에 로컬 저장이 적합합니다. 또한 서버 저장 시 불필요한 저장 공간과 비용이 발생할 수 있어 로컬 저장 방식을 선택했습니다.

# 2. 인증 및 보안

## 🔐 인증 관련 질문

### 1. 로그인/로그아웃 동작
**Q: 로그인은 어떻게 처리하나요?**
- **답변**: Supabase Auth를 사용하여 이메일/비밀번호 기반 인증을 구현했습니다. 로그인 성공 시 세션 토큰이 생성되어 로컬에 저장되며, 이후 API 요청 시 자동으로 인증 정보가 포함됩니다.

**Q: 로그아웃 시 어떤 데이터가 삭제되나요?**
- **답변**: 로그아웃 시 Supabase 세션만 삭제되며, 서버에 저장된 리뷰나 즐겨찾기 데이터는 삭제되지 않습니다. 로컬의 SharedPreferences에 저장된 검색 기록, 알림 기록 등도 유지됩니다.

**Q: 비로그인 사용자도 앱을 사용할 수 있나요?**
- **답변**: 네, 비로그인 사용자도 지도 조회, 장소 검색, 리뷰 읽기 등은 가능합니다. 다만 리뷰 작성, 즐겨찾기 추가 등은 로그인이 필요합니다.

---

### 2. 데이터 보안 및 권한 관리
**Q: 다른 사용자의 리뷰를 수정하거나 삭제할 수 있나요?**
- **답변**: 아니요, 불가능합니다. Supabase의 RLS(Row Level Security) 정책을 통해 사용자는 자신이 작성한 리뷰만 수정/삭제할 수 있습니다. 다른 사용자의 리뷰는 읽기만 가능합니다.

**Q: RLS 정책은 어떻게 설정했나요?**
- **답변**: 
  - **reviews 테이블**: 사용자는 자신의 리뷰만 INSERT/UPDATE/DELETE 가능, 모든 사용자가 SELECT 가능
  - **favorites 테이블**: 사용자는 자신의 즐겨찾기만 SELECT/INSERT/DELETE 가능
  - **places 테이블**: 모든 사용자가 SELECT 가능, 인증된 사용자가 INSERT/UPDATE 가능

**Q: API 키나 인증 정보가 노출될 위험이 있나요?**
- **답변**: Supabase의 Anon Key는 클라이언트에 노출되지만, RLS 정책을 통해 서버 측에서 권한을 제어하므로 안전합니다. Service Role Key는 서버에서만 사용되며 클라이언트에는 노출되지 않습니다.

---

### 3. 사용자 정보 관리
**Q: 사용자 프로필 정보는 어디에 저장되나요?**
- **답변**: 기본 사용자 정보(이메일, ID)는 Supabase Auth에 저장되며, 추가 프로필 정보(닉네임, 아바타 등)는 `profiles` 테이블에 저장됩니다.

**Q: 사용자 정보를 수정할 수 있나요?**
- **답변**: 현재는 프로필 수정 기능이 제한적입니다. 향후 닉네임, 프로필 이미지 수정 기능을 추가할 예정입니다.

---

### 4. 세션 관리
**Q: 로그인 세션은 얼마나 유지되나요?**
- **답변**: Supabase Auth의 기본 세션 유지 시간을 따릅니다. 세션이 만료되면 자동으로 로그아웃되며, 다시 로그인해야 합니다.

**Q: 여러 기기에서 동시에 로그인할 수 있나요?**
- **답변**: 네, 가능합니다. 각 기기마다 독립적인 세션이 생성되며, 한 기기에서 로그아웃해도 다른 기기의 세션은 유지됩니다.

---

## 🔒 보안 관련 질문

### 1. 데이터 무결성
**Q: 리뷰 데이터가 조작되거나 삭제될 위험이 있나요?**

**Q: 악의적인 사용자가 가짜 리뷰를 대량으로 작성할 수 있나요?**
- **답변**: 현재는 제한이 없지만, 향후 Rate Limiting이나 CAPTCHA 등을 추가하여 대량 작성 방지 기능을 구현할 예정입니다.

### 2. 개인정보 보호
**Q: 사용자의 위치 정보는 어떻게 처리하나요?**
- **답변**: 위치 정보는 기기에서만 사용되며, 서버에 저장되지 않습니다. 현재 위치 기반 검색이나 알림 기능에만 사용됩니다.

**Q: 리뷰에 포함된 개인정보는 어떻게 보호하나요?**
- **답변**: 리뷰 작성 시 사용자 ID는 저장되지만, 이메일 등 개인정보는 노출되지 않습니다. 사용자는 자신의 리뷰만 수정/삭제할 수 있습니다.

# 3. 기능 동작 방식

## 🗺️ 지도 및 마커 관련 질문

### 1. 지도 마커 표시
**Q: 지도에 표시되는 마커는 어떻게 생성되나요?**
- **답변**: Supabase에서 리뷰 데이터를 조회하여 각 장소별 평균 소음 수준을 계산하고, 그에 따라 색상이 다른 마커를 동적으로 생성합니다. Optimal(녹색), Good(노란색), Normal(주황색), Loud(빨간색)로 구분됩니다.

**Q: 마커는 실시간으로 업데이트되나요?**
- **답변**: 현재는 화면에 진입하거나 지도를 이동할 때마다 데이터를 새로 조회합니다. 실시간 업데이트는 Supabase Realtime 기능을 활용하면 가능하지만, 현재는 구현되지 않았습니다.

**Q: 지도에 표시되는 장소는 어떻게 필터링되나요?**
- **답변**: 사용자가 선택한 소음 수준 필터에 따라 마커를 표시/숨김 처리합니다. 또한 현재 화면 영역(뷰포트) 내의 장소만 표시하여 성능을 최적화했습니다.

---

### 2. 소음 측정 기능
**Q: 소음 측정은 어떻게 동작하나요?**
- **답변**: Android의 `AudioRecord` API를 사용하여 마이크로부터 오디오 신호를 실시간으로 읽어옵니다. 신호의 진폭을 계산하여 dB 값을 산출하며, 50ms마다 UI를 업데이트합니다.

**Q: 소음 측정의 정확도는 어떤가요?**
- **답변**: 현재는 신호 처리 기반의 기본적인 측정 방식을 사용합니다. 마이크 캘리브레이션이나 A-weighting 필터는 적용되지 않아, 절대적인 정확도보다는 상대적인 비교에 유용합니다.

**Q: 측정된 dB 값은 수정할 수 있나요?**
- **답변**: 리뷰 작성 시에는 수정할 수 있지만, 리뷰 저장 후에는 수정할 수 없습니다. 데이터의 신뢰성을 유지하기 위한 정책입니다.

---

### 3. 검색 기능
**Q: 장소 검색은 어떻게 동작하나요?**
- **답변**: 카카오 로컬 API를 사용하여 키워드 기반 장소 검색을 수행합니다. 검색 결과는 리스트로 표시되며, 선택 시 지도로 이동합니다.

**Q: 검색 기록 자동완성은 어떻게 동작하나요?**
- **답변**: SharedPreferences에 저장된 최근 검색어를 기반으로 자동완성 제안을 표시합니다. 검색어 입력 중 일치하는 검색어가 있으면 드롭다운으로 제안합니다.

**Q: 검색 결과는 캐싱되나요?**
- **답변**: 현재는 검색 기록만 저장되며, 검색 결과 자체는 캐싱되지 않습니다. 향후 메모리 캐싱 기능을 추가할 예정입니다.

---

### 4. 리뷰 관리 기능
**Q: 리뷰 작성 시 이미지는 어떻게 처리하나요?**
- **답변**: 갤러리에서 선택하거나 카메라로 촬영한 이미지를 Supabase Storage에 업로드합니다. 업로드된 이미지의 URL을 JSON 배열 형식으로 리뷰 데이터에 저장합니다.

**Q: 리뷰 수정 시 소음 수준은 수정할 수 있나요?**
- **답변**: 아니요, 소음 수준(dB 값)은 수정할 수 없습니다. 별점과 텍스트만 수정 가능하며, 이는 데이터의 신뢰성을 유지하기 위한 정책입니다.

**Q: 리뷰 삭제 시 이미지는 어떻게 되나요?**
- **답변**: 현재는 리뷰 삭제 시 Supabase Storage의 이미지 파일은 자동으로 삭제되지 않습니다. 향후 이미지 정리 기능을 추가할 예정입니다.

---

### 5. 알림 시스템
**Q: 조용한 장소 추천 알림은 어떻게 동작하나요?**
- **답변**: WorkManager를 사용하여 주기적으로 백그라운드 작업을 실행합니다. 사용자 위치 기반으로 주변 조용한 장소를 탐지하고, 발견 시 알림을 발송합니다.

**Q: 알림은 언제 발송되나요?**
- **답변**: WorkManager의 PeriodicWorkRequest를 통해 주기적으로 실행됩니다. 정확한 주기는 설정에 따라 다르며, 사용자가 알림 설정을 끄면 발송되지 않습니다.

**Q: 소음 임계값 알림은 어떻게 동작하나요?**
- **답변**: 사용자가 설정한 소음 임계값을 초과하는 장소에 대한 리뷰가 작성되면 알림을 발송합니다. 즐겨찾기한 장소에 대해서만 동작합니다.

---

### 6. 날씨 정보
**Q: 날씨 정보는 어디서 가져오나요?**

**Q: 날씨 정보는 캐싱되나요?**

# 4. 기술 선택 및 아키텍처

## 🛠️ 기술 스택 선택 관련 질문

### 1. Firebase vs Supabase
**Q: 왜 Firebase 대신 Supabase를 선택했나요?**
- **답변**: 
  - **오픈소스**: Supabase는 오픈소스 기반으로 자체 호스팅이 가능합니다
  - **PostgreSQL**: 강력한 관계형 데이터베이스 기능 제공
  - **RLS 정책**: Row Level Security를 통한 세밀한 권한 제어
  - **비용**: 무료 플랜이 더 관대함
  - **개발 경험**: SQL 기반 쿼리가 더 직관적

**Q: Supabase의 단점은 없나요?**
- **답변**: 
  - Firebase에 비해 생태계가 작아서 자료나 예제가 상대적으로 적음
  - 실시간 기능(Realtime)이 Firebase보다 제한적일 수 있음
  - 하지만 프로젝트 요구사항에는 충분히 적합함

---

### 2. 지도 SDK 선택
**Q: 왜 카카오맵을 선택했나요?**
- **답변**: 
  - **한국 시장**: 한국 사용자에게 친숙하고 정확한 장소 정보 제공
  - **무료 플랜**: 개발 단계에서 충분한 무료 할당량 제공
  - **API 품질**: 카카오 로컬 API와의 통합이 용이함

**Q: Google Maps를 사용하지 않은 이유는?**
- **답변**: 카카오맵이 한국 지역의 장소 정보가 더 상세하고, 카카오 로컬 API와의 통합이 자연스러워서 선택했습니다. 또한 무료 할당량도 충분했습니다.

---

### 3. 아키텍처 패턴
**Q: MVVM 패턴을 사용했나요?**
- **답변**: 부분적으로만 사용했습니다. Repository 패턴은 구현했지만, ViewModel은 사용하지 않았습니다. Activity에서 직접 Repository를 호출하는 구조입니다.

**Q: ViewModel을 사용하지 않은 이유는?**
- **답변**: 프로젝트 규모가 크지 않아서 ViewModel의 이점(화면 회전 시 데이터 유지 등)보다는 단순한 구조가 더 적합하다고 판단했습니다. 향후 확장 시 ViewModel을 도입할 수 있습니다.

**Q: Jetpack Compose를 사용하지 않은 이유는?**
- **답변**: XML 레이아웃이 더 익숙하고, 팀원들의 학습 곡선을 고려하여 XML을 선택했습니다. 또한 기존 코드베이스와의 일관성을 유지하기 위함입니다.

---

### 4. 데이터베이스 선택
**Q: 로컬 데이터 저장은 어떻게 하나요?**
- **답변**: 
  - **Supabase**: 서버 데이터베이스로, 모든 사용자가 공유하는 영구 데이터 저장 (리뷰, 즐겨찾기 등)
  - **SharedPreferences**: 로컬 저장소로, 간단한 설정값, 검색 기록, 알림 기록 등 키-값 형태의 작은 데이터 저장
  - 현재는 서버 기반 데이터 저장을 주로 사용하며, 향후 오프라인 모드 지원을 위해 로컬 데이터베이스 도입을 검토 중입니다

**Q: SharedPreferences만으로 충분한가요?**
- **답변**: 현재 프로젝트에서는 SharedPreferences로 충분합니다. 검색 기록, 알림 기록 등 간단한 데이터는 SharedPreferences로 관리하고 있으며, 복잡한 관계형 데이터는 모두 Supabase에 저장합니다. 향후 오프라인 캐싱이 필요하면 Room Database나 다른 로컬 데이터베이스를 도입할 수 있습니다.

---

### 5. 네트워크 라이브러리
**Q: Retrofit 대신 Ktor Client를 선택한 이유는?**
- **답변**: 
  - **Kotlin Multiplatform**: Ktor는 Kotlin Multiplatform을 지원하여 향후 확장성 좋음
  - **코루틴 네이티브**: 코루틴을 네이티브로 지원하여 비동기 처리 용이
  - **경량**: Retrofit보다 가볍고 설정이 간단함
  - **Supabase SDK**: Supabase SDK가 내부적으로 Ktor를 사용하여 일관성 유지

---

### 6. 이미지 처리
**Q: Coil 라이브러리를 선택한 이유는?**
- **답변**: 
  - **경량**: Glide나 Picasso보다 가볍고 빠름
  - **코루틴 지원**: Kotlin 코루틴을 네이티브로 지원
  - **간단한 API**: 사용법이 직관적이고 간단함
  - **캐싱**: 자동으로 메모리 및 디스크 캐싱 지원

---

## 🏗️ 아키텍처 관련 질문

### 1. Repository 패턴
**Q: Repository 패턴을 사용하는 이유는?**
- **답변**: 
  - **데이터 소스 추상화**: 데이터 소스를 쉽게 전환 가능
  - **테스트 용이성**: Mock Repository로 단위 테스트 가능
  - **유지보수성**: 데이터 접근 로직을 한 곳에 모아 관리

**Q: Repository는 어떻게 구현했나요?**
- **답변**: 현재는 Repository 패턴을 사용하지 않고, Activity에서 직접 Supabase를 호출하는 구조입니다. 향후 확장성을 고려하여 Repository 패턴 도입을 검토 중입니다.

---

### 2. 비동기 처리
**Q: 비동기 처리는 어떻게 하나요?**
- **답변**: Kotlin Coroutines를 사용합니다. `lifecycleScope.launch`를 통해 Activity 생명주기에 맞춰 코루틴을 실행하고, `withContext(Dispatchers.IO)`로 네트워크 작업을 백그라운드 스레드에서 처리합니다.

**Q: 에러 처리는 어떻게 하나요?**
- **답변**: try-catch 블록으로 에러를 처리하고, 사용자에게 Toast 메시지로 알립니다. 네트워크 에러, 인증 에러 등 다양한 예외 상황을 구분하여 처리합니다.

---

### 3. 의존성 주입
**Q: 의존성 주입은 어떻게 하나요?**
- **답변**: 현재는 Singleton 패턴을 사용합니다 (SupabaseManager). Koin이나 Hilt 같은 DI 라이브러리는 사용하지 않았습니다. 프로젝트 규모가 작아서 Singleton으로 충분했습니다.

# 5. 성능 최적화 및 한계점

## ⚡ 성능 최적화 관련 질문

### 1. 지도 성능
**Q: 지도에 많은 마커가 있을 때 성능 문제는 없나요?**
- **답변**: 뷰포트 기반 필터링을 구현했습니다. 현재 화면에 보이는 영역 내의 장소만 마커로 표시하여 성능을 최적화했습니다. 또한 카메라 이동 시 디바운싱을 적용하여 불필요한 API 호출을 줄였습니다.

**Q: 마커 클러스터링은 구현했나요?**
- **답변**: 현재는 구현되지 않았습니다. 마커가 많을 때 클러스터링을 통해 성능을 개선할 수 있지만, 현재는 뷰포트 필터링으로 충분히 처리 가능합니다. 향후 필요 시 추가할 예정입니다.

---

### 2. 네트워크 최적화
**Q: API 호출을 최소화하기 위한 방법은?**
- **답변**: 
  - 뷰포트 기반 데이터 로딩 (필요한 영역만 조회)
  - 디바운싱 적용 (카메라 이동 시)
  - 캐싱 (검색 기록, 이미지 캐싱)
  - 하지만 검색 결과나 리뷰 데이터는 아직 메모리 캐싱이 구현되지 않았습니다

**Q: 이미지 로딩 최적화는 어떻게 하나요?**
- **답변**: Coil 라이브러리가 자동으로 메모리 및 디스크 캐싱을 처리합니다. 또한 썸네일을 사용하거나 이미지 크기를 조정하는 기능은 향후 추가할 예정입니다.

---

### 3. 메모리 관리
**Q: 메모리 누수 방지는 어떻게 하나요?**
- **답변**: 
  - View Binding 사용으로 null 안전성 확보
  - lifecycleScope를 통한 생명주기 관리
  - 코루틴 취소 처리
  - 하지만 LeakCanary 같은 도구는 아직 도입하지 않았습니다

**Q: 대용량 이미지 처리 시 문제는 없나요?**
- **답변**: 현재는 원본 이미지를 그대로 업로드하고 표시합니다. 대용량 이미지의 경우 메모리 부족이나 느린 로딩이 발생할 수 있습니다. 향후 이미지 리사이징이나 썸네일 생성 기능을 추가할 예정입니다.

# 6. 기타 질문

## 🎯 프로젝트 목표 및 배경

### 1. 프로젝트 동기
**Q: 이 프로젝트를 시작하게 된 동기는 무엇인가요?**
- **답변**: 조용한 작업 공간을 찾는 수요가 증가하고 있지만, 기존 리뷰 앱들은 소음 정보가 부족하거나 주관적 평가에 의존합니다. 실제 측정 데이터 기반의 객관적 소음 정보를 제공하여 사용자들이 더 나은 선택을 할 수 있도록 돕고자 했습니다.

**Q: 타겟 사용자는 누구인가요?**
- **답변**: 조용한 공간이 필요한 사람들 (재택근무자, 학생, 집중 작업이 필요한 사람 등)을 타겟으로 합니다.

---

### 2. 제안서 대비 변경사항
**Q: 제안서와 실제 구현의 차이점은?**
- **답변**: 
  - **추가된 기능**: 날씨 정보 연동, Noise Trend 그래프, 검색 기록, 알림 시스템
  - **기술 변경**: Firebase → Supabase로 변경
  - **미구현 기능**: 오프라인 모드, 실시간 업데이트, 페이징 등

**Q: 변경한 이유는?**
- **답변**: Supabase가 오픈소스이고 PostgreSQL 기반으로 더 강력한 기능을 제공하며, RLS 정책을 통한 세밀한 권한 제어가 가능하기 때문입니다.
